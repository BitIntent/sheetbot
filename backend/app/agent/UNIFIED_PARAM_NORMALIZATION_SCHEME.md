# 统一参数规范化方案

## 问题分析

### 当前问题
1. **字段名格式不一致**：LLM 返回字段名（字符串）或列号（数字），导致前端无法处理
2. **范围参数格式不匹配**：后端传递 `{ startRow, startCol, endRow, endCol }`，前端期望 `{ start: { row, col }, end: { row, col } }`
3. **修复方式分散**：每个工具都需要单独修复，维护成本高

### 根本原因
- **缺乏统一的参数契约**：前后端对参数格式的理解不一致
- **缺乏统一的转换层**：参数转换逻辑分散在各个工具中
- **缺乏类型安全**：没有统一的类型检查和转换机制

## 解决方案：统一参数规范化架构

### 方案概述

建立一个**三层参数规范化架构**：
1. **参数契约层**：定义统一的参数格式规范
2. **规范化层**：在参数传递边界统一转换
3. **适配层**：前端统一适配不同格式的参数

### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    LLM 输出（各种格式）                        │
│  - 字段名: ["销售人员", "产品ID"]                            │
│  - 列号: [7, 2]                                             │
│  - 范围: { startRow: 1, startCol: 1, ... }                │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│           后端：统一参数规范化层（Normalization Layer）        │
│                                                              │
│  1. 参数类型检测                                             │
│  2. 格式识别（字段名 vs 列号）                                │
│  3. 统一转换（列号 → 字段名，需要访问 Excel 状态）            │
│  4. 格式标准化（范围参数统一格式）                            │
│  5. 输出标准格式参数                                          │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│           标准参数格式（Standard Parameter Format）          │
│                                                              │
│  - 字段名：统一为字符串数组 ["销售人员", "产品ID"]            │
│  - 范围：统一为 { start: { row, col }, end: { row, col } } │
│  - 类型：明确标注参数类型                                     │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│           前端：统一参数适配层（Adaptation Layer）            │
│                                                              │
│  1. 接收标准格式参数                                         │
│  2. 向后兼容旧格式（自动转换）                                │
│  3. 类型验证                                                 │
│  4. 传递给具体工具函数                                       │
└─────────────────────────────────────────────────────────────┘
```

## 详细方案

### 方案1：后端统一规范化（推荐）

#### 核心思想
在**后端工具函数输出前**，统一规范化所有参数，确保传递给前端的参数格式一致。

#### 实现要点

**1. 建立参数规范化中心（Parameter Normalization Center）**

位置：`backend/app/agent/param_normalizer.py`

功能：
- **字段名规范化**：检测列号，转换为字段名（需要访问 Excel 状态）
- **范围参数规范化**：统一范围参数格式
- **类型验证**：确保参数类型正确
- **向后兼容**：支持多种输入格式

**2. 参数规范化流程**

```
输入参数（LLM 输出）
    ↓
类型检测（字段名 vs 列号）
    ↓
格式识别（范围参数格式）
    ↓
统一转换（列号 → 字段名，范围格式标准化）
    ↓
类型验证（确保格式正确）
    ↓
输出标准格式参数
```

**3. 字段名转换机制**

需要访问 Excel 状态（表头信息）：
- 从 `excel_state` 中获取表头
- 将列号映射到字段名
- 支持相对列号和绝对列号

**4. 范围参数标准化**

统一转换为：
```python
{
    "start": {"row": 1, "col": 1},
    "end": {"row": 201, "col": 12}
}
```

**5. 工具函数集成**

所有工具函数在返回操作前调用规范化函数：
```python
# 伪代码
normalized_params = normalize_operation_params(operation["params"], excel_state)
operation["params"] = normalized_params
```

#### 优点
- ✅ **统一处理**：所有参数规范化逻辑集中在一处
- ✅ **类型安全**：统一类型检查和转换
- ✅ **易于维护**：修改一处，所有工具受益
- ✅ **向后兼容**：支持多种输入格式

#### 缺点
- ⚠️ **需要访问 Excel 状态**：字段名转换需要表头信息
- ⚠️ **可能增加延迟**：规范化过程需要时间

#### 实施步骤
1. 扩展 `param_normalizer.py`，添加字段名转换和范围参数标准化
2. 在 `excel_tools.py` 中，所有工具函数调用规范化函数
3. 测试所有工具，确保参数格式一致

---

### 方案2：前端统一适配层

#### 核心思想
在**前端接收参数后**，统一适配不同格式的参数，确保工具函数接收到的参数格式一致。

#### 实现要点

**1. 建立参数适配中心（Parameter Adaptation Center）**

位置：`frontend/src/utils/paramAdapter.js`

功能：
- **字段名适配**：检测列号，转换为字段名（使用表头信息）
- **范围参数适配**：统一范围参数格式
- **类型验证**：确保参数类型正确
- **向后兼容**：支持多种输入格式

**2. 参数适配流程**

```
接收参数（后端输出）
    ↓
格式检测（字段名 vs 列号，范围格式）
    ↓
统一适配（列号 → 字段名，范围格式标准化）
    ↓
类型验证（确保格式正确）
    ↓
传递给工具函数
```

**3. 工具函数集成**

所有工具函数在执行前调用适配函数：
```javascript
// 伪代码
const adaptedParams = adaptOperationParams(operation.params, workbook)
operation.params = adaptedParams
```

#### 优点
- ✅ **前端已有表头信息**：转换方便
- ✅ **不需要修改后端**：后端代码不变
- ✅ **统一处理**：所有参数适配逻辑集中在一处

#### 缺点
- ⚠️ **前端复杂度增加**：需要维护适配逻辑
- ⚠️ **可能重复处理**：如果后端已经规范化，前端再次适配

#### 实施步骤
1. 创建 `paramAdapter.js`，实现统一的参数适配逻辑
2. 在 `excelOperations.js` 的 `executeOperation` 函数中调用适配函数
3. 测试所有工具，确保参数格式一致

### 方案3：混合方案（最佳实践）

#### 核心思想
**后端规范化 + 前端适配**，双重保障，确保参数格式一致。

#### 实现要点

**1. 后端：基础规范化**
- 统一范围参数格式
- 类型检查和基本转换
- 不进行字段名转换（因为需要 Excel 状态）

**2. 前端：完整适配**
- 字段名转换（使用表头信息）
- 范围参数格式适配
- 类型验证和错误处理

**3. 参数契约**
- 定义清晰的参数格式规范
- 后端输出标准格式
- 前端适配标准格式

#### 优点
- ✅ **双重保障**：后端规范化 + 前端适配
- ✅ **职责分离**：后端处理基础规范化，前端处理业务适配
- ✅ **向后兼容**：支持多种输入格式

#### 缺点
- ⚠️ **实现复杂度较高**：需要同时维护两套逻辑

## 推荐方案：方案1（后端统一规范化）

### 为什么推荐方案1？

1. **单一职责**：参数规范化是后端的职责
2. **统一处理**：所有工具函数统一调用规范化函数
3. **易于维护**：修改一处，所有工具受益
4. **类型安全**：统一类型检查和转换

#### 2. 工具函数集成

所有工具函数在返回操作前调用规范化函数：

```python
# 伪代码示例
async def create_pivot_table(args: Dict[str, Any]) -> Dict[str, Any]:
    # ... 现有逻辑 ...
    
    # 规范化参数
    normalized_params = normalize_operation_params(
        operation["params"],
        excel_state=current_excel_state,  # 需要传递 Excel 状态
        operation_type="create_pivot_table"
    )
    operation["params"] = normalized_params
    
    return result
```

#### 3. Excel 状态传递

需要修改工具函数调用方式，传递 Excel 状态：

```python
# 在 excel_agent.py 中
async def handle_tool_call(tool_name, args, excel_state):
    # 调用工具函数，传递 excel_state
    result = await tool_function(args, excel_state=excel_state)
    return result
```

#### 4. 参数契约定义

定义清晰的参数格式规范：

```python
# 标准参数格式
STANDARD_PARAM_FORMATS = {
    "field_names": {
        "type": "array",
        "items": {"type": "string"},
        "description": "字段名数组，必须是字符串"
    },
    "range": {
        "type": "object",
        "properties": {
            "start": {"type": "object", "properties": {"row": int, "col": int}},
            "end": {"type": "object", "properties": {"row": int, "col": int}}
        },
        "description": "范围参数，统一格式"
    }
}
```

### 实施优先级

1. **第一阶段**：实现范围参数规范化（不需要 Excel 状态）
2. **第二阶段**：实现字段名规范化（需要 Excel 状态）
3. **第三阶段**：所有工具函数集成规范化函数
4. **第四阶段**：测试和优化

### 风险评估

**风险1：需要访问 Excel 状态**
- **影响**：字段名转换需要表头信息
- **解决方案**：在工具函数调用时传递 Excel 状态

**风险2：性能影响**
- **影响**：规范化过程可能增加延迟
- **解决方案**：缓存规范化结果，优化转换逻辑

**风险3：向后兼容**
- **影响**：现有代码可能依赖旧格式
- **解决方案**：支持多种输入格式，逐步迁移

## 总结

### 核心原则

1. **统一规范化**：在参数传递边界统一处理
2. **类型安全**：统一类型检查和转换
3. **向后兼容**：支持多种输入格式
4. **易于维护**：集中管理，避免分散修复

### 推荐实施路径

1. **短期**：实现范围参数规范化（方案1的第一阶段）
2. **中期**：实现字段名规范化（方案1的第二阶段）
3. **长期**：建立完整的参数契约和验证机制

### 预期效果

- ✅ **彻底解决**：字段名格式不一致和范围参数格式不匹配问题
- ✅ **统一维护**：所有工具函数统一调用规范化函数
- ✅ **易于扩展**：新增工具函数自动受益
- ✅ **类型安全**：统一类型检查和转换机制
